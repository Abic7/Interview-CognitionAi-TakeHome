❓ WHAT: The Problem

The Core Issue: Remediation Bottleneck

Detection is solved. Remediation is broken.

Every organization runs security scanners (Snyk, Trivy, pip-audit, OWASP). They find vulnerabilities accurately and quickly. But then what?

Manual Remediation Process:
  1. Read the issue: "CVE-2026-25645 in requests 2.31.0"
  2. Understand impact: "Does this affect us?"
  3. Research the fix: "What version is safe?"
  4. Write code: Find files, update versions, test
  5. Validate: Run tests, check regressions
  6. Deploy: Open PR, wait for review, merge

Time per issue: 20-30 minutes

Key Architectural Decisions

1

Minimal Orchestrator (150 lines)

Decision: Keep the orchestrator extremely simple — just trigger Devin, poll for results, record status.

What We Rejected:
  • Full-stack React frontend (complexity)
  • Task queues (Celery, Bull)
  • Webhook systems (requires public endpoint)
  • Complex state machines
Why This Works: Devin is the decision-maker. Orchestrator's only job is send task + watch results. No complex logic needed.
2

Polling Instead of Webhooks

Decision: Every 30 seconds, check GitHub for new PRs instead of waiting for webhook callbacks.

What We Rejected:
  • Devin webhooks (not in v3 API)
  • GitHub webhooks (requires public endpoint)
  • Long-polling (wasted resources)
Why This Works: Simple, reliable, works locally. 30-second latency is fine for issues that took weeks to fix.
3

SQLite for State Management

Decision: Single-file database to track findings and PR status.

What We Rejected:
  • In-memory state (lost on restart)
  • PostgreSQL (overkill, requires external service)
  • Redis (unnecessary complexity)
Why This Works: Lightweight, persistent, observable, Docker-friendly. Perfect for single-instance orchestrator.
4

One Session for Multiple Issues

Decision: Give Devin all issues at once, let it work through them.

What We Rejected:
  • One session per issue (wastes setup time)
  • New sessions when one completes (fragmented context)
Why This Works: Devin understands context across issues. Can parallelize internally. Cheaper (fewer API calls).

The Real Difference: Autonomous Reasoning

Devin isn't just executing steps. It's:

1. Understanding Context — "This codebase uses requests for API calls. Rate limiting is critical."

2. Anticipating Problems — "Upgrading might break rate limiting. Let me check the changelog."

3. Solving Problems — "The param changed. Let me update the code AND improve it for security."

4. Validating Solutions — "Run tests... check warnings... verify it works."

5. Taking Initiative — "Actually, this config should be better. Let me improve it."

A bot can't do any of that. It can only follow rules. Devin reasons through each unique situation because the work is too varied for rules.

Phase 3: Closed-Loop Automation

Devin doesn't just fix, Devin audits:
  1. 1. Devin runs security audits (pip-audit, bandit, trivy)
  2. 2. Devin creates GitHub issues for each finding
  3. 3. Devin fixes the issues automatically
  4. 4. Devin validates: "CI passed? All checks green?"
  5. 5. Devin auto-merges (if configured)
  6. 6. Orchestrator reports: "127 → 43 vulnerabilities in 2 hours"
  7. 7. Daily cron: Repeat automatically

Result: Vulnerability backlog never accumulates. Issues found and fixed within hours, not weeks.
